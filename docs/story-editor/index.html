Good typesetting is invisible; great typesetting is felt. From justified columns to careful kerning, every detail affects the final impression on the page.

What this demonstrates: Text is shaped with HarfBuzz (the same engine Scribus uses) to resolve ligatures and compute glyph advances. Style runs are extracted from a DOM tree (like Scribus's TextShaper::itemizeStyles()), each run shaped separately. A greedy line-breaking algorithm with hyphenation wraps and justifies text to fit the box. Each line is rendered as an SVG <text> element with styled <tspan> elements.

Features: OpenType ligatures (fi, fl, ffi, ffl) via HarfBuzz, English hyphenation via Knuth-Liang patterns, full justification, two-column flow, and bold/italic/bold-italic styled runs.

Comparison with Scribus text layout algorithm
Aspect This demo Scribus (pageitem_textframe.cpp)
Styled text model DOM tree walked to produce style runs — each run has text + {bold, italic}. Like Scribus's TextShaper::itemizeStyles() Per-character ScText objects (each inherits CharStyle), collapsed into runs by style before shaping
Font handling Variable font (EB Garamond) loaded once per axis file (upright + italic). Bold uses font.setVariations({wght:700}) on the same binary. Same buffers registered as @font-face so HarfBuzz and browser metrics match exactly Fonts managed by SCFonts registry. Each style variant is a separate ScFace with its own HarfBuzz hb_font_t. Supports variable fonts via hb_font_set_variations() with user-specified axis values
Shaping HarfBuzz WASM, one shape call per style run with wght variation axis set for bold (700). Glyph streams stitched per paragraph HarfBuzz (native), text itemized by BiDi level, script, style, and OT features — each run shaped separately
Line breaking Greedy, single-pass — break at last space or soft hyphen when line overflows Also greedy single-pass (not Knuth-Plass). Records break opportunities at spaces and ScLayout_HyphenationPossible flags, then uses the best one when line is full
Hyphenation Pre-inserted soft hyphens via Knuth-Liang patterns (hyphen JS library). All break points computed up front Hyphenation points flagged during shaping phase onto GlyphCluster objects. Supports consecutive-hyphen limit, manual soft hyphens, and per-style enable/disable
Justification Single strategy: distribute extra space evenly across inter-word gaps Multi-priority: (1) shrink spaces & extend glyphs to min ratio, (2) extend glyphs to max ratio, (3) insert CJK implicit spaces & extend word spaces, (4) letter spacing. Controlled by min/max word & glyph extension settings
Glyph positioning Browser handles final glyph placement via <text> + @font-face; HarfBuzz used only for width measurement HarfBuzz provides exact x/y offsets and advances; Scribus places each glyph precisely in a GlyphBox within a box tree hierarchy (LineBox → GlyphBox)
BiDi / scripts None — LTR Latin only Full Unicode BiDi algorithm, per-script itemization, RTL reordering
Columns / flow Two equal-width columns — text flows from first to second at the midpoint (like linked frames) N-column layout within a single frame, plus linked frames across pages. Text flows automatically when a column or frame fills up
Frame features Fixed-width rectangle, no indents or wraps Drop caps, baseline grid, optical margins, text-wraps around objects, first/left/right line indents, column gap control
Caching None — reshapes on every update ShapedTextCache per-paragraph; only edited paragraphs are re-shaped

Both use the same fundamental approach: itemize by style → HarfBuzz shapes each run → greedy line fill → justify → flow across columns. The difference is depth — Scribus handles the full complexity of professional typesetting (mixed scripts, frame geometry, optical margins, glyph-level stretch) while this demo isolates the core loop in ~300 lines of JavaScript.
